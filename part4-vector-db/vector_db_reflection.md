## Vector DB Use Case

A traditional keyword-based search over 500-page legal contracts would struggle to answer natural language questions like “What are the termination clauses?” reliably. Keyword search mainly matches exact words or simple variations, so it might miss relevant sections that use different phrasing such as “early exit conditions” or “agreement dissolution terms.” It also cannot easily understand context; if the word “termination” appears in a completely different context, keyword search may rank that result highly even though it is not about ending the contract. As contracts are long and dense, a lawyer would need to manually skim many pages of loosely related hits.

A vector database changes this by storing semantic embeddings of sentences, paragraphs, or clauses. When a lawyer asks a question in plain English, the system converts the query into an embedding and finds contract chunks with similar meaning, not just similar words. This allows it to return sections that discuss ending the agreement even if they never use the exact query terms. The vector database is typically used alongside a more traditional document store: the raw contracts remain in a standard storage system, while their embeddings live in the vector DB for fast similarity search. Together with a language model, the vector index enables question-answering over long contracts, surfacing the most relevant passages and allowing the AI to generate focused, human-readable summaries for the lawyer.
