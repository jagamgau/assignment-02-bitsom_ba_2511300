## Architecture Recommendation

For the food delivery startup described, I would recommend a Data Lakehouse architecture rather than a pure Data Warehouse or a pure Data Lake. The startup collects many different kinds of data: GPS location logs from drivers, free-text customer reviews, payment transactions, and restaurant menu images. A traditional data warehouse works well for structured tables such as transactions and aggregated metrics, but it is not designed to store raw images, high-volume logs, or unstructured text efficiently. A pure data lake can store all of these files cheaply in their native formats, but by itself it does not provide strong schema management or fast SQL analytics on top.

A Lakehouse combines the strengths of both. All raw data can land in low-cost object storage in its original form: JSON logs for GPS events, text files for reviews, parquet tables for transactions, and binary image files for menus. On top of this storage, the Lakehouse maintains table-like metadata and schemas so that analysts can run SQL queries directly over curated views of the data. This makes it possible to build dashboards for order volume, delivery time, and restaurant performance, while still keeping the detailed historical data available for future machine learning use cases such as ETA prediction or review sentiment analysis. The Lakehouse approach also simplifies governance because there is a single storage layer instead of duplicating data separately in a warehouse and a lake. For a fast-growing startup with diverse data types and evolving analytics needs, this flexibility and unification is a major advantage.
