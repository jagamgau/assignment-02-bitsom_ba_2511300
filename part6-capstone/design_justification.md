## Storage Systems

In this architecture I use different storage systems for different goals so that each workload runs on technology that fits its needs. The core patient records, diagnoses, medications, and billing events live in an OLTP relational database (for example MySQL or Postgres). This database is optimized for frequent small reads and writes and provides strong ACID guarantees, which is critical when doctors are updating charts and orders in real time. ICU monitoring devices send a high-volume stream of vitals, so those measurements are ingested through a streaming layer (such as Kafka) into a time-series or stream-analytics store that can handle high throughput and support alert rules over recent data. For analytics and reporting, I use an analytical storage layer (data warehouse or lakehouse) that holds cleaned, historical data from the OLTP system and billing feeds. This store is designed for large aggregations and complex BI queries. Finally, unstructured documents such as discharge summaries and doctor notes are processed by an embedding service and stored as vectors in a vector database. The vector DB allows semantic search over patient text so that doctors can ask questions in plain English and retrieve the most relevant passages.

## OLTP vs OLAP Boundary

The OLTP–OLAP boundary in my design sits between the operational patient database and the downstream analytical storage. The EHR / EMR system and billing system interact directly with the OLTP patient DB, which is the system of record for current patient state and must remain highly available and strongly consistent. No analytical workloads or heavy reports run directly on this database. Instead, changes from the OLTP DB are moved through an ETL or streaming layer into the analytical warehouse. Once data crosses this boundary, it is treated as read-mostly and slightly delayed (for example, updated hourly or daily), which is acceptable for management dashboards and model training. Analytical consumers such as the readmission risk model and the management BI tools query only the warehouse, not the live transactional system. The doctor query UI uses the vector DB for text search but can still call the OLTP DB for the latest critical facts, keeping queries that need fresh data on the transactional side.

## Trade-offs

A significant trade-off in this design is the added complexity and cost of maintaining multiple specialized systems instead of a single consolidated database. Running an OLTP store, a streaming platform, a data warehouse, a vector DB, and a time-series store introduces more moving parts, more failure modes, and higher operational overhead. However, trying to force all workloads into one system would either compromise safety (by running heavy analytics on the OLTP DB) or limit functionality (for example, losing semantic search or real-time alerts). I would mitigate this trade-off by standardizing on managed cloud services where possible, using infrastructure-as-code and monitoring to keep deployments repeatable and observable. I would also define clear data contracts and schemas between the OLTP layer, ETL pipelines, and analytical storage, so that each team understands where their responsibilities start and end. With good automation and governance, the benefits of performance, scalability, and clinical safety justify the extra complexity.
